## Goal

Deliver the tooling that moves the existing SQLite kanban board into Postgres,
plus the runbook to cut over — **without** executing a live cutover (Phase 5 stays
human-driven) and **without** changing the default backend or touching the live
single-writer dispatch path.

Deliverables:
1. **`hermes_cli/kanban/migrate_sqlite_to_pg.py`** — read-only export → transform →
   load → verify, with `--dry-run` and `--execute` modes.
2. **`plans/kanban-postgres-migration/cutover-runbook.md`** — the Phase-5 runbook,
   gating on the `phase-3-tail` items as BLOCKING preconditions.
3. **Tests** under `tests/hermes_cli/kanban/`, reusing the docker-PG conformance fixture.

## The central transform: per-board-file → single-DB-with-`board`-column

SQLite stores **each board as a separate DB file**; the PG schema is **one
database** with a `board TEXT` column on every table. So the migrator stamps
`board=slug` onto every row it loads.

### The global-IDENTITY-id hazard (decided: single-board guard)

Four tables — `task_comments`, `task_events`, `task_runs`,
`kanban_profile_wake_events` — are `INTEGER PRIMARY KEY AUTOINCREMENT` in SQLite
(per-board, each file starts at 1) but **`BIGINT GENERATED BY DEFAULT AS IDENTITY`
with `PRIMARY KEY (id)` ALONE in Postgres** — i.e. *globally* unique across boards.
Loading more than one board into the single PG database would collide on those PKs
**and** on every column that references them: `task_events.run_id`,
`tasks.current_run_id`, `kanban_profile_event_claims.event_id`,
`kanban_notify_subs.last_event_id`, `kanban_profile_event_subs.last_event_id`,
`kanban_profile_wake_events.claimed_event_cursor`.

**Decision (live reality is one board): single-board guard.** The migrator
enumerates boards via `kanban_db.list_boards()` and, if it finds **more than one**,
**refuses loudly** (no silent partial load). For the single board it loads all ids
**1:1** into an empty target, then reseqs the sequences. True multi-board id-remap
is deferred until a second board actually exists. `t_…` TEXT task ids and epoch
(`BIGINT`) timestamps are preserved unconditionally (the PG `tasks` PK is
`(board, id)`, so even a cross-board `t_…` collision would be fine).

## Component breakdown (each independently testable)

| Unit | Responsibility |
|---|---|
| `enumerate_boards()` | `kanban_db.list_boards()` → single-board guard (refuse if >1) |
| `read_source(board)` | Open the board DB **read-only** (read-only URI connection as in `kanban_board_doctor._quick_check` — **not** `immutable`, so a live rehearsal read stays correct); read all 9 migrated tables into row dicts |
| `validate(rows)` | Pre-flight: every TEXT column is valid UTF-8; `task_events.payload` / `task_runs.metadata` parse as JSON. Collect **all** offenders → **abort + report** (no silent drop/coerce). |
| `load(conn, schema, board, rows)` | FK-parent-first batched `executemany`, stamping `board=slug`, JSON→JSONB, inside **one transaction** |
| `reseq(conn, schema)` | `setval(pg_get_serial_sequence(table,'id'), max(id), true)` for the 4 IDENTITY tables (or `(…, 1, false)` when empty) |
| `verify(source, conn, schema, board)` | The verification stack (below); returns a structured diff |
| `dry_run(dsn, board)` / `execute(dsn, board, force)` | Orchestration (below) |
| `main()` | argparse CLI |

## The two modes

- **`--dry-run`** → `CREATE SCHEMA kanban_dryrun_<token>` (token derived from
  board slug + source-file mtime, **not** a wall clock — keeps re-runs stable and
  is deterministic), set `search_path` to it, apply `pg_schema.sql`, load + verify,
  then **`DROP SCHEMA … CASCADE`**. Zero residue; runs against the **real** target
  server so it surfaces real permission/type/constraint/JSON issues.
- **`--execute`** → guard: if the board already has rows in the target (`public`)
  schema, **abort** unless `--force` is given (which `DELETE`s that board's rows in
  FK order first). Then load + `reseq` + **in-transaction** structural verify, and
  only then **COMMIT** (all-or-nothing). Post-commit, run the store-parity read.
- The migrator uses its **own** psycopg connection (autocommit off) for load+verify
  so a failed/garbage load never partially commits. The store-parity read uses a
  `PostgresKanbanStore`.
- **Load order** (parent-first; PG has **no** FK constraints, so order is cosmetic
  but kept for clarity): `tasks → task_runs → task_events → task_comments →
  task_links → kanban_notify_subs → kanban_profile_event_subs →
  kanban_profile_event_claims → kanban_profile_wake_events`. `task_runs` precedes
  `task_events` because `task_events.run_id` points at it.
- **Sidecars:** `kanban_notifier_heartbeats` and metrics snapshots are **not**
  migrated (ephemeral; start fresh in PG).

## Verification stack

Run in dry-run, and as the post-load gate in `--execute`:

1. **Row counts** source-vs-target, all 9 migrated tables, **exact**.
2. **PG referential integrity** (SQL — the schema enforces none): no orphan
   `task_links` (parent/child must exist in `tasks`), no orphan
   `task_comments` / `task_events` / `task_runs` / `kanban_notify_subs` /
   `kanban_profile_event_subs` / `kanban_profile_wake_events` (`task_id` must exist),
   `task_events.run_id` and `tasks.current_run_id` resolve to `task_runs`,
   `kanban_profile_event_claims.event_id` resolves to `task_events`.
3. **Id / sequence**: per-IDENTITY-table `max(id)` matches source; reseq leaves
   `nextval = max+1`.
4. **Store-parity** (the conformance lens): a sample of tasks read via
   `SqliteKanbanStore`(source) vs `PostgresKanbanStore`(target) must be field-equal;
   `list_tasks` length matches; spot-check `list_comments` / `list_runs` for a few
   tasks. Proves *semantics*, not just bytes.
5. **Source health precondition**: run the existing SQLite `run_board_doctor` on the
   source first; surface critical issues before loading.

Any mismatch → structured diff + **nonzero exit**. A dedicated test deliberately
breaks a load to prove the verifier *fails* (so the gate is trustworthy).

### Small additive dependency

The store-parity read against the dry-run schema needs the `PostgresKanbanStore`'s
pool connections to `SET search_path` to that schema. Add an **optional**
`search_path` (or `configure`) kwarg to `pg_pool.make_pool` — fork-owned, additive,
default behavior unchanged.

## Bad-data handling (decided: fail loud)

The pre-flight `validate()` pass detects non-UTF-8 TEXT and unparseable JSON,
collects **all** offenders, and **aborts with a report** before any write. It does
**not** silently drop or coerce. The human scrubs the source with the existing
util (there is precedent — `kanban.db.pre_utf8_scrub_*` backups) and re-runs. This
realizes the parent design's "no more torn non-UTF-8 rows" by *catching* them at
migration time rather than letting Postgres reject mid-load.

## CLI surface (decided: standalone module)

```
venv/bin/python -m hermes_cli.kanban.migrate_sqlite_to_pg \
    (--dry-run | --execute) [--force] [--dsn DSN] [--board SLUG] [--json] [--report PATH]
```

A one-shot ops tool → standalone module, **not** wired into the daily
`hermes kanban` CLI. Mode (`--dry-run` | `--execute`) is **required** (no default).
DSN resolves via `pg_pool.resolve_dsn()` (`HERMES_KANBAN_PG_DSN` → `kanban.postgres.dsn`)
when `--dsn` is omitted. Board resolves via `list_boards()` (single-board guard)
when `--board` is omitted.

## Cutover runbook (`cutover-runbook.md`) — Phase 5, human-driven

Ordered steps:
1. **Preconditions gate** — all `phase-3-tail` items closed (BLOCKING; see below);
   Supabase DSN provisioned; maintenance window scheduled.
2. **Dry-run** against the real target → verification green.
3. **Quiesce:** `hermes gateway stop`.
4. **Final export + load:** `--execute` into the provisioned Postgres → verify.
5. **Flip config:** `kanban.backend=postgres` + set `HERMES_KANBAN_PG_DSN` /
   `kanban.postgres.dsn`.
6. **Restart** gateway + dashboard → smoke-verify (list tasks, create/claim/complete
   on a throwaway task, dashboard loads).
7. **Rollback** (valid only before PG takes divergent writes): flip
   `kanban.backend=sqlite`, restart. The SQLite file is retained untouched.

### `phase-3-tail` items — BLOCKING preconditions (Phase 4 documents, does not close)

Tracked in marker commits `a69ef9949` / `157d4dab3` and the `kanban-pg-phase3-glue`
memory note; loud `# phase-3-tail:` markers in `store_postgres.py` / `kanban_glue.py`:
- PG `detect_crashed_workers` reap-registry clean-exit (rc=0) protocol-violation
  classification.
- PG pre-spawn validation auto-block emit.
- PG host-local `SIGTERM→grace→SIGKILL` kill-ladder (currently delegated to an
  injected `signal_fn`) + reconcile store `signal_fn(pid,sig)` vs glue
  `enforce_runtime_kill` ladder.
- Systemic-spawn-failure SIBLING pre-emptive block in the glue.
- (cleanup) dead gateway `_kanban_advance` / `_rewind` / `_unsub` / `_profile_*`
  helpers now unused by the notifier.
- Confirm whether non-single-writer sqlite is a supported production config (affects
  which of the above are blocking).

These are runtime `PostgresKanbanStore`/glue correctness gaps, orthogonal to moving
data; they get their own gated follow-up before the live cutover.

## Test plan (sketch — detailed in the implementation plan)

Reuse the docker-PG conformance fixture (`tests/hermes_cli/kanban/conftest.py`):
- **Round-trip:** seed a fixture SQLite board (tasks/links/comments/events/runs/
  subs/claims/wake-events) → migrate into docker-PG (throwaway schema) → assert
  counts, integrity, max-ids, sequence reseq, and store-parity equality.
- **Multi-board guard refuses:** two board files → migrator aborts.
- **Idempotency:** second `--execute` refuses; with `--force` it succeeds and counts
  still match.
- **Bad data:** invalid-JSON payload / non-UTF-8 text → migrator aborts with a clear
  report, **no partial write**.
- **Dry-run leaves no residue:** throwaway schema created then dropped.
- **Verifier catches corruption:** deliberately skip a table / drop a row → verify
  fails with nonzero exit (proves the gate).

## Boundaries (restated)

- Default backend stays `sqlite`; the live single-writer dispatch path is untouched.
- The migrator reads SQLite **read-only**; it never mutates a source board DB.
- **No live cutover** in Phase 4 — Phase 5 needs a provisioned Supabase DSN, explicit
  go-ahead, and a maintenance window.

## Success criteria

- `migrate_sqlite_to_pg.py` migrates the single live board into a throwaway PG schema
  (dry-run) with the full verification stack green.
- Re-runnability, multi-board-guard, and bad-data behaviors are covered by tests, and
  the verifier is proven to fail on injected corruption.
- `cutover-runbook.md` documents the quiesce→export→load→flip→restart→verify flow
  with rollback and the `phase-3-tail` BLOCKING preconditions.
- Default backend unchanged; no edits to the live dispatch path; source DBs untouched.
